The real cost

What do data anomalies cost your team?

Every undetected anomaly is a downstream failure waiting to surface in a standup, an audit, or a board meeting.

Stale data, silent failures

A table stops updating. No alert fires. Stakeholders make decisions on data that is hours or days out of date before anyone notices.

Volume spikes no one caught

A row count doubles overnight. The pipeline passes. The anomaly propagates downstream before your team sees the dashboard break.

Schema drift breaks pipelines

A column gets renamed upstream. Your downstream transforms fail silently. You spend the morning tracing a breakage that happened three deploys ago.

Value violations reach production

Null rates spike. Regex patterns break. Distribution shifts go undetected until a data consumer files a ticket and the trust conversation starts over.

The solution

A better approach to data anomaly detection

DataHub monitors your datasets continuously, learns what normal looks like, and surfaces deviations before they become incidents your team has to explain.

Smart Assertions learn your data

ML-powered assertions observe historical patterns and set dynamic thresholds automatically. No manual rule-writing required to get started.

  • Dynamic thresholds based on historical baselines
  • Learns seasonality and expected variance patterns
  • Reduces false positives without manual tuning

Four assertion types, one platform

Freshness, volume, schema, and field-level value assertions run continuously across every monitored dataset in your platform.

  • Freshness and volume checks on every ingestion run
  • Schema change detection with column-level diff
  • Null rate, regex, and distribution value checks

Automated incidents, end to end

When an assertion fails, DataHub opens an incident automatically. When the data recovers, the incident closes. Your team works the queue, not the noise.

  • Incidents open and close without manual intervention
  • Route alerts to Slack, PagerDuty, or webhooks
  • Full audit trail linked to lineage and ownership

Scale monitoring with domain rules

Write one monitoring rule and apply it to every dataset in a domain. Coverage scales with your platform without proportional engineering effort.

  • Domain-scoped rules propagate to new datasets
  • Tag-based targeting for tier-one critical assets
  • Manage thousands of assertions from one interface

Common questions

Frequently asked questions about detecting anomalies

DataHub detects four categories of data anomalies: freshness failures (tables that stop updating on schedule), volume anomalies (row counts that spike or drop outside expected ranges), schema drift (column additions, removals, or type changes), and field-level value violations (null rate changes, regex pattern failures, and distribution shifts). Smart Assertions use ML to set dynamic thresholds for freshness and volume checks, while schema and value assertions can be configured manually or auto-suggested based on observed patterns.
Smart Assertions learn the historical behavior of each dataset before setting thresholds. This means they account for known seasonality, expected batch sizes, and regular schema evolution patterns. Thresholds adjust over time as data patterns change, which reduces the alert fatigue that comes from static rule-based systems. You can also tune sensitivity per dataset or per domain if your environment has specific requirements.
DataHub supports anomaly detection across 80+ data sources including Snowflake, BigQuery, Redshift, Databricks, dbt, Airflow, Looker, Tableau, and Spark. Coverage depends on the connector and what metadata each source exposes. Freshness and volume checks are available for most warehouse and lake sources. Schema change detection is available wherever DataHub ingests structural metadata. Field-level value checks require query access to the underlying data.
When an assertion fails, DataHub opens an incident and can route notifications to Slack channels, PagerDuty, or any webhook endpoint your team already uses. Incidents include links to the affected dataset, the assertion that failed, and the lineage context showing downstream impact. When the underlying data recovers and the assertion passes again, the incident closes automatically. Your team does not need to manually track resolution state.
Yes. Domain-scoped monitoring rules let you write one assertion template and apply it to every dataset within a domain. New datasets that match the domain criteria inherit the rule automatically, so coverage grows with your platform without proportional engineering effort. Tag-based targeting lets you apply stricter thresholds to tier-one critical assets while keeping lighter coverage on lower-priority datasets. This approach is designed for organizations managing hundreds to thousands of monitored assets.
